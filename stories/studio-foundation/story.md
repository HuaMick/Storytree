# The studio foundation

**Outcome —** An operator reviews the project record through one browsable forum studio.

apps/studio is a hand-built, single-process Vite dev app (run with `pnpm --filter studio dev`) that turns the repo's own docs/ corpus and a synthesised guidance Library into a reviewable forum: read rendered ADRs/glossary, anchor comments onto exact text spans / sections / whole topics and resolve them, and browse-author-seed a categorised Library of injectable guidance artifacts. The whole 'backend' is one Vite middleware file (server/devApi.ts) serving docs read-only from <repo>/docs and persisting comments + assets to git-tracked JSON stores under apps/studio/data. HONESTY: the app compiles and runs, but package.json has only dev/build/preview/typecheck — zero test tooling, no *.test/*.spec files anywhere in the package, no scripted UAT. Every capability below is a RETROSPECTIVE spec over already-working code; its contracts describe the isolated tests that WOULD prove each behaviour (citing real code at file:line) and its UAT describes the walkthrough that WOULD prove it against the real running app. The lone capability with proof automatable TODAY is the Library seeder (a standalone Node script whose output is directly assertable). Nothing here is 'proven', 'healthy', or 'mapped'; proof status is authored-only.

## Dependency graph (UAT-generated)

Edges are **generated by UAT need**, never hand-drawn: A → B means A's walkthrough
requires B to be real (ADR-0002). The graph is acyclic; `dev-server-persistence-backbone`
and `seed-library-corpus` are the roots.

- `read-corpus` → `dev-server-persistence-backbone`
  - The read UAT (browse the index, open a doc, follow a cross-link) needs the real /api/* middleware mounted before Vite's SPA fallback — read-corpus owns its doc handlers but only RIDES that namespace-routing seam, which is the persistence backbone's configureServer registration.
- `annotate-topic` → `dev-server-persistence-backbone`
  - The annotate UAT posts a comment and reloads to re-find it; that requires POST /api/comments → writeStore and the post-reload GET → readStore round-trip the backbone guarantees.
- `annotate-topic` → `read-corpus`
  - The annotate UAT must open a real rendered corpus document (markdown with slugged headings) to select a span and attach a section/text anchor — there is no topic to annotate until read-corpus renders one.
- `resolve-comment` → `dev-server-persistence-backbone`
  - The resolve UAT clicks Resolve/Reopen, requiring the PATCH /api/comments?id round-trip to stamp/clear resolvedAt in comments.json and the refreshComments GET re-fetch to read it back across surfaces — exactly the backbone's persist+read seam. (annotate-topic is NOT an edge: the UAT seeds the target comment directly, so the create flow is never exercised.)
- `browse-library` → `dev-server-persistence-backbone`
  - The browse UAT loads the Library grid and an artifact detail from GET /api/assets and the doc index from GET /api/docs — both served by the backbone's /api/* middleware over the on-disk JSON store.
- `browse-library` → `seed-library-corpus`
  - The browse UAT asserts the live per-category counts (definition 53, pattern 11, guardrail 8, principle 5, techstack 4) and opens a seeded artifact with a real doc: citation — that 81-artifact corpus only exists because the seeder produced assets.json.
- `browse-library` → `read-corpus`
  - The browse UAT's final leg follows an artifact's doc: reference to LAND on the cited ADR rendered in-app (#/doc/<relpath>) — resolving and rendering that doc is read-corpus's journey.
- `author-library-artifact` → `dev-server-persistence-backbone`
  - The author UAT creates, edits, then deletes an artifact and inspects assets.json on disk each time — the durable POST/PATCH/DELETE /api/assets mutations and createdAt/updatedAt stamping are the backbone's persistence guarantee.
- `author-library-artifact` → `browse-library`
  - The author UAT's delete leg routes back to the Library list and confirms the artifact is gone, and the create/edit legs land on the rendered artifact detail (AssetView) — the browse-and-view surface author mutations return into.
